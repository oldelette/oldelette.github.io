command

command module 是個可以在遠端上執行指令的指令模組 (Commands Modules), 剛入門 Ansible 會用 module 不多？
只要 Linux Shell 會通的指令都可以透過它使用. 但它不支援變數 (variables) 和 <, >, |, ; 和 & 等運算, 若有這類需求請改用 shell module

command module source code: link

stat module 用於檢查指定目錄是否存在, 結果會存儲在 dir_stat 中.
然後使用 when 條件句, 只有當目錄存在 (dir_stat.stat.exists) 時才會執行 rmdir 命令

---
- name: Play with Ansible
  hosts: vboxservers
  tasks:
    - name: Check if directory exists
      ansible.builtin.stat:
        path: /home/gavin/0107
      register: dir_stat         

    - name: Remove directory if exists
      ansible.builtin.command:
        cmd: rmdir -p /home/gavin/0107
      when: dir_stat.stat.exists

stat

stat module 是用來檢查檔案狀態的檔案模組 (Files Modules)。其類似的 Linux 指令為 stat。

檢查檔案是否存在, 若不存在則建立它.

---
- name: Play with Ansible
  hosts: vboxservers
  tasks:
    - name: check the stat target exists
      stat:
        path: /home/gavin/stat
      register: stat_vimrc
    
    - name: touch vimrc
      file:
        path: /home/gavin/stat
        state: touch
        mode: "u=rw,g=r,o=r"
      when: stat_vimrc.stat.exists == false

Ansible

學習 Ansible

Ansible 的系統架構:

  • Ansible的部署模式不需要依靠代理程式 (agent), 與 Puppet 的拉取（Pull model）屬性相比, Ansible 的屬性則為推播式（Push model)
  • Ansible 不需要透過架設額外的 Master 主機來主導部署過程（除非使用 Ansible Tower), Ansible 只需透過SSH協定, 即可對遠端伺服器進行控制或部署, 而這樣的設計, 因為不必為所有管理節點預先安裝代理程式, 因而方便了使用者.

Introduction

ANSIBLE 的優點

  • 簡化的自動化作業
    Ansible 是一個簡單易用的平台,易於安裝與設定,學習速度非常快.
    在不到 30 分鐘的時間內,就能安裝及設定系統,並針對伺服器執行臨時命令,以解決特定問題
  • 低學習曲線
    Ansible 非常容易部署,因為它不使用代理程式或額外的自訂安全性基礎架構.
    它也運用 YAML,這種簡單的語言可透過方針來說明自動化工作.
    方針會在清單中定義的主機上推送所需的設定,甚至可以臨時執行（透過命令列，而不需要在檔案中定義）
  • 立即自動化
    可以透過 Ansible ping 主機的那一刻,就能開始將環境自動化.從小型工作開始,遵循最佳實務做法,優先處理可為企業創造更多價值、解決重大問題及節省時間的工作,並提高生產力。

CHEF vs Puppet vs Ansible vs Terraform

架構程式碼的實施工具

Ansible 比起其他自動化部屬的工具有兩大特點：

  1. 在遠端只需要 python 環境與 ssh server，不需要其他 agent 也沒有額外的自定義安全基礎設施, 因此易於部署.
  2. 它使用 YAML, 以 Ansible Playbook 的形式, 允許您以接近英語的方式描述您的自動化作業.

  • Puppet 是很傳統的主從模式（master-slave）架構.
  • Ansible 是不需要代理人（agent）的
  • Terraform 則是較新一代的 IaC 工具, 它比較重視程式碼所代表的雲端服務生命週期（Cloud Service Lifecycle）

Puppet

Puppet 是主從式架構（Master Slave architecture, 所有配置的體現方式（manifests）都是存在 Master Server 上, 每個 server 節點上都會裝一個 agent.

在節點上的 agent 會向 master 要求 catalog 去部署節點伺服器
這個叫做 Pull Model（拉取模式）

  • Puppet 算是「宣示式」的架構程式碼
    Puppet catalog 是一個描述被部署節點最終狀態的文檔.
    它包括了在節點上需部署的軟體，, 各模組間的相關連性.
  • 每個部署節點上的 agent 會把伺服器節點的特定資訊（Facts）傳回給 Master, Master 會根據這些 Facts 去生出該節點的 catalog, 再把 catalog 傳回給節點上的 agent, 再由 agent 在節點上安裝軟體、配置參數.
  • Agent 安裝完畢後,會把結果傳回 Master node 上的報告收集程式（Report Collector, 作為審核日誌

Puppet 的配置架構，相對簡單.如下圖所示, Puppet Manifest 是用來記錄雲端服務所有節點需要安裝的程式及其次序, 這也就是雲端服務的部署腳本

Terraform

Differences Between Terraform and Ansible link

從架構面來看，Terraform 與 Puppet 或 Ansible 相似, 它仍是一個宣示式的架構程式碼編譯工具, 可以高效能地部署、更新、版本化基礎設施和雲端服務, 具有多層次的資源管理功能, 從上層的 SaaS 配置到底層的 IaaS 配置, 都可以使用 Terraform 統一進行管理.

  • Terraform Core
    TF core 負責把 TF-config 讀入, 與 project 的現狀（state）比較
    然後查詢 Terraform 平台上 provider 的 plugin, 以產生一個 Terraform Action Plan, 在相關 Provider 的伺服器上執行

  • Terraform config file
    每個專案會用 HCL (HashiCorp Configuration Language) 撰寫 一個 TF config file, 來描述一個服務.
    Terraform 特別的地方, 就是它支援很多很多不同的 Providers, 包括了 IaaS、PaaS、utilities、SaaS.
    所以 TF config file 通常可以作為雲端服務的 藍圖（blueprint), 也可以用來做版控.

  • Project State
    Terraform 的另一個特色, 是它會根據上次部署的紀錄log, 把服務配置的狀態存在內部.

  • Providers
    Terraform 支援非常多的私有雲與公有雲服務供應商（Providers）

Terraform： 適合建立和管理基础設施, 尤其是涉及跨多層平台的情况.
Ansible： 更適合配置管理、應用程式部署和通用的自動化任務,尤其在網路設備的自動化方面具有廣泛的支持

Ansible Terminology

Ansible 的術語

  • Modules
    透過 Ansible 推送到各個節點的腳本稱為 “module”, Ansible 預設會透過 ssh 來執行 module.
    通常你會透過現有的 module 來達成需求, 但必要時也可以自己開發 module, 開發的語言有 python、ruby、bash 等等.
    許多相關的 Module 集合起來會變成 Collection,例如 Azure.Azcollection.

  • Module utilities
    當多個 module 使用相同的程式碼時, Ansible 將這些函數存儲為 module utilities, 以最大限度地減少重複和維護.
    module utilities 只能用 python 或 powershell 開發.

  • Plugins
    用來擴充 Ansible 的核心功能,如轉換資料、記錄輸出、連接到 inventory 等.
    Plugin 與 Module 不同的點在於 Module 定義了一個介面讓使用者自行設定一些參數,再根據輸入來對遠端做相應的操作；
    Plugin 主要執行在 Control Node,用來處理 Ansible 核心的輸入和輸出.

  • Inventory
    Ansible 在一個 INI、YAML 等紀錄它管理的 inventory, 該文件讓你幫機器的 ip 分組, 格式類似：

    






    [webservers]
www1.example.com
www2.example.com

[dbservers]
db0.example.com
db1.example.com

  • Playbooks
    Playbook 可以精細地編排 (orchestrate) 基礎設設施,並非常詳細地控制一次要處理多少台機器.
    一個基礎設施可能持續使用很多年,因此 Ansible 盡量不推出新的特殊語法或功能,讓你不用為了維護久遠以前的基礎設施而花很多時間複習 Ansible 語法.
    下方是一個簡單的 playbook：

    










    ---
- hosts: webservers
  serial: 5 # update 5 machines at a time
  roles:
  - common
  - webapp

- hosts: content_servers
  roles:
  - common
  - content

  • Ansible search path
    如果您編寫自己的 Modules、Plugins 來擴展 Ansible 的核心功能, 可能在 Ansible 控制節點的不同位置有多個名稱相似或相同的檔案.
    search path 決定了 Ansible 將在指定的 playbook 使用哪些文件.
    當 Ansible 找到給定運行中包含的每個 playbook 和 role 時，它會將自動將可能有關的目錄附加到搜索路徑中,並自動載入找到的檔案.

Install

$ pip install ansible

$ pip install ansible-lint

Example













---
- name: Play with Ansible
  hosts: localhost
  tasks:
    - name: Just execute df -h
      ansible.builtin.command: df -h
      register: output
      changed_when: output.rc == 0

    - name: Show stdout
      ansible.builtin.debug:
        msg: "{{ output.stdout_lines }}"
  • 第 2 行：names 是這個 play 的名稱,命名為 “Play with Ansible”
    在此範例中, name、hosts、tasks 被稱為 play keywords, Ansible 讀到這些 keyword 會做相應的動作
  • 第 3 行：hosts 是這個 play 的布屬對象 (target), 以逗點隔開的 hostname 或是 ip 位址.
  • 第 4 行：tasks 是部屬對象所要執行的任務
  • 第 5 行：自訂任務的名稱. name、register、changed_when 被稱為 task keywords
  • 第 6 行：ansible.builtin.command 是一個 Ansible 內建的 module, 它在所有布屬對象執行一個指令, 在範例中我讓 localhost 執行 df -h
  • 第 7 行：register 會接收這個 task 的回傳訊息（即 df -h 的 stdout）並指派給 “output” 變數.以寫程式類比的話，相當於宣告了一個名為 output 的變數來儲存 df -h 的輸出
  • 第 8 行：changed_when 是特別用來表示 targets 執行完 task 後，狀態有沒有被「改變」，狀態是否被改變的判斷條件是由撰寫 playbook 的人自訂的.
    在這個範例中定義當 df -h 的回傳值為 0 時即代表機器的狀態被改變了.
    如果這個 task 完全不會影響機器的狀態, 可以把這行改為 change_when: false
    如果不寫這一行，Ansible 也能正常執行, 不過 change_when 就永遠會是 true, 而且會被 ansible-lint 要求補上這一行.
  • 第 9 行：第二個任務
  • 第 10 行：用內建的 debug module 來輸出訊息到終端機
  • 第 11 行：輸出 df -h 的每一行 stdout

virtualbox 設定參考

首先準備兩個 linux 主機，我是用 Virtual Box 裝兩台 Linux server, 並設定網路使用 NAT-Network (類似於 NAT, 差別在於 vm 之間可以溝通)
本機要安裝 sshpass 才能進行接下來的操作

inventory.ini

在目錄中新增一個名為 inventory.ini 的檔案：














; [vboxhost]
; localhost ansible_user=gavin ansible_password=0710

[vboxservers]
alpine1 ansible_host=172.168.1.200 

[allservers:children]
vboxservers
; vboxhost

[vboxservers:vars]
ansible_user=gavin
ansible_password=0710
  • [vboxhost]
    定義了一個名為 vboxhost 的節點組, 裡面包含了本地主機（localhost）的 Ansible 連接資訊.
    在這裡指定了 ansible_user 和 ansible_password，以便在 Ansible 執行時使用.
    (本地主機上執行 Ansible 任務不需要這樣的設置)
  • [vboxservers]
    這是另一個節點組, 包含了一台名為 alpine1 的伺服器.
    它定義了該伺服器的 ansible_host(IP address), 同時也設置了 ansible_user 和 ansible_password 變數, 用於與server 的連接
  • 第 11 行透過 :vars 來將重複使用的變數值一次指派給整個群組的機器

在這個檔案中我將兩台虛擬機分配在 vboxservers、本機分配在 localservers
且透過 ansible_host、ansible_user、ansible_password 來設定 ssh 連線的變數
常我們不會直接用 ansible_password 儲存密碼, 而是用 ansible vault 或其他第三方軟體來保護密碼

ansible.cfg

在同樣的目錄中新增一個 ansible.cfg，複製以下內容，讓 ansible 登入陌生的機器時自動儲存 finger print

[defaults]
host_key_checking = False

nsible.cfg 是 Ansible 的設定檔案之一
用於設置 Ansible 的各種運行參數和默認行為.
在 [defaults] 部分設定 host_key_checking = False 的話, 會關閉 SSH 主機金鑰檢查, 這意味著當 Ansible 第一次連接到一台新的主機時, 不會提示用戶確認主機金鑰.
這樣可以避免在連接新主機時出現交互式提示, 使得自動化腳本在連接新主機時不會被阻擾.
